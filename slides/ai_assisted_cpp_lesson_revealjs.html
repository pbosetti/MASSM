AI-Assisted C++ Development for Engineers

Designing first, generating second

Why this topic matters in mechatronics

  • Coordinate handling and geometric utilities
  • Sensor-data processing pipelines
  • Calibration and test-bench software
  • Robotics and automation support tools
  • Reusable numerical components inside larger systems

Many engineering tools are built from small reusable software components. A class such as Vector2D is simple enough to understand in one lesson, but rich enough to discuss design choices, correctness, testing, and AI-assisted workflow.

Core principle

We decide

  • requirements
  • interface
  • constraints
  • acceptance criteria

AI proposes

  • candidate code
  • alternatives
  • tests
  • explanations

We verify

  • review
  • compilation
  • numerical checks
  • targeted tests

This is the whole lesson in one slide. Requirements, interfaces, assumptions, and validation criteria remain a human responsibility.

Learning outcomes

By the end of the lesson, students should be able to:

  • define requirements for a small C++ component
  • design a minimal public interface
  • use AI in bounded, controllable steps
  • review generated code critically
  • validate correctness with tests and numerical checks

This is not a lesson on advanced C++ techniques. It is a lesson on disciplined workflow.

What AI does well

Code generation

  • boilerplate
  • repetitive operators
  • small utilities

Support work

  • test scaffolding
  • documentation drafts
  • compiler-diagnostic explanations

Review support

  • alternative formulations
  • refactoring suggestions
  • first-pass code review hints

AI is useful for acceleration, not for authorship of the engineering intent.

What AI does poorly

Design judgment

  • inventing requirements
  • choosing abstractions
  • deciding public API boundaries

Technical judgment

  • numerical reasoning
  • hidden assumptions
  • subtle domain mistakes

Process discipline

  • scope control
  • signaling uncertainty
  • stopping when evidence is weak

Plausible-looking code is not the same as trustworthy code.

AI strengths and weaknesses

Useful acceleration

AI is effective for:

  • boilerplate
  • repetitive operators
  • test skeletons
  • code explanations

Typical risks

AI often introduces:

  • extra features
  • hidden assumptions
  • style inconsistency
  • fragile numeric checks

Human role

The engineer must retain control of:

  • requirements
  • interface design
  • acceptance criteria
  • final validation

Use this slide to reinforce the operational division of responsibility.

Example problem

Implement a Vector2D class representing a 2D point/vector.

Required operations:

  • construction
  • accessors
  • addition
  • subtraction
  • scalar multiplication
  • norm
  • distance
  • printing

The example is deliberately small so the workflow stays visible.

Constraints

  • Simple and readable
  • Suitable for beginners
  • No dynamic allocation
  • No inheritance
  • Easy to test
  • Clear separation of interface and implementation

Constraints reduce ambiguity. Without them, AI tends to invent scope and style.

Design before code

Questions to answer before implementation:

  • Should data be public or private?
  • Why use const?
  • Member or non-member printing?
  • Exact equality or tolerance?
  • What belongs in the public interface?

This is where engineering authorship begins. Design decisions precede syntax.

Proposed public interface

class Vector2D {
public:
    Vector2D();
    Vector2D(double x, double y);

    double x() const;
    double y() const;

    double norm() const;
    double distanceTo(const Vector2D& other) const;

    Vector2D operator+(const Vector2D& other) const;
    Vector2D operator-(const Vector2D& other) const;
    Vector2D operator*(double scalar) const;

private:
    double x_;
    double y_;
};

std::ostream& operator<<(std::ostream& os, const Vector2D& v);

The interface is the contract. AI should implement this contract, not rewrite it unless asked.

Why this interface is reasonable

  • Private representation limits accidental misuse
  • Small API reduces the error surface
  • Arithmetic operators improve readability
  • const clarifies intent and enables safer use
  • Stream output integrates with standard C++ I/O

Translate each design choice into a practical benefit.

First AI prompt: generate the header

Generate a simple C++ header for a class named Vector2D with private double
members x_ and y_. Provide a default constructor, a constructor with x and y,
const accessors, norm(), distanceTo(), operators +, -, scalar multiplication by
double, and a non-member stream output operator. Keep it minimal and
beginner-friendly. Do not add templates, comparisons, normalization, exceptions,
or advanced features.

A good prompt is mainly a well-bounded specification.

Review the generated header

Checklist:

  • Did AI add unwanted features?
  • Are methods const where appropriate?
  • Are members private?
  • Are includes minimal?
  • Is the interface still ours?

Model rejection openly. Removing AI-generated extras is part of competent use.

Second AI prompt: generate the implementation

Generate the .cpp implementation for this Vector2D interface. Use std::sqrt for
norm and distance. Keep the code readable for engineering students with limited
C++ background. Avoid unnecessary abstractions.

Once the interface is frozen, implementation can be delegated more safely.

Review the implementation

Check for:

  • correct formulas
  • clear naming
  • minimal includes
  • consistent operator semantics
  • no hidden complexity

This is where line-by-line review should be presented as normal engineering practice.

Reference implementation

#ifndef VECTOR2D_H
#define VECTOR2D_H

#include <iosfwd>

class Vector2D {
public:
    Vector2D();
    Vector2D(double x, double y);

    double x() const;
    double y() const;

    double norm() const;
    double distanceTo(const Vector2D& other) const;

    Vector2D operator+(const Vector2D& other) const;
    Vector2D operator-(const Vector2D& other) const;
    Vector2D operator*(double scalar) const;

private:
    double x_;
    double y_;
};

std::ostream& operator<<(std::ostream& os, const Vector2D& v);

#endif

Present this as an accepted baseline, not the only possible design.

Reference implementation (.cpp)

#include "Vector2D.h"

#include <cmath>
#include <ostream>

Vector2D::Vector2D() : x_(0.0), y_(0.0) {}

Vector2D::Vector2D(double x, double y) : x_(x), y_(y) {}

double Vector2D::x() const {
    return x_;
}

double Vector2D::y() const {
    return y_;
}

double Vector2D::norm() const {
    return std::sqrt(x_ * x_ + y_ * y_);
}

double Vector2D::distanceTo(const Vector2D& other) const {
    const double dx = x_ - other.x_;
    const double dy = y_ - other.y_;
    return std::sqrt(dx * dx + dy * dy);
}

Vector2D Vector2D::operator+(const Vector2D& other) const {
    return Vector2D(x_ + other.x_, y_ + other.y_);
}

Vector2D Vector2D::operator-(const Vector2D& other) const {
    return Vector2D(x_ - other.x_, y_ - other.y_);
}

Vector2D Vector2D::operator*(double scalar) const {
    return Vector2D(x_ * scalar, y_ * scalar);
}

std::ostream& operator<<(std::ostream& os, const Vector2D& v) {
    os << "(" << v.x() << ", " << v.y() << ")";
    return os;
}

Example usage

#include "Vector2D.h"
#include <iostream>

int main() {
    Vector2D a(3.0, 4.0);
    Vector2D b(1.0, 2.0);

    std::cout << "a = " << a << '\n';
    std::cout << "b = " << b << '\n';
    std::cout << "a + b = " << (a + b) << '\n';
    std::cout << "a - b = " << (a - b) << '\n';
    std::cout << "2a = " << (a * 2.0) << '\n';
    std::cout << "norm(a) = " << a.norm() << '\n';
    std::cout << "distance(a,b) = " << a.distanceTo(b) << '\n';

    return 0;
}

Compile with warnings

g++ -std=c++17 -Wall -Wextra -pedantic main.cpp Vector2D.cpp -o vector_demo
  • Warnings are part of review
  • Generated code is not finished if diagnostics are sloppy
  • Build cleanliness is one acceptance criterion

Compilation is not just “does it build”.

What could go wrong

  • Missing const
  • Too many features
  • Public data without reason
  • Poor header design
  • Exact float comparisons
  • Unclear operator semantics

The class is small, but the failure modes are real and inspectable.

Validation mindset

Question:

What would convince us that this class is correct enough for use?

Possible evidence:

  • known reference cases
  • readable implementation
  • clean compilation
  • targeted tests
  • explicit numerical assumptions

Move from coding to evidence.

AI prompt for tests

Generate a small C++ test program for Vector2D using <cassert>. Test default
construction, addition, subtraction, scalar multiplication, norm of (3,4), and
distance between known points. Use a floating-point tolerance for numeric
comparisons.

AI is often more valuable for generating test scaffolding than for inventing software design.

Reference test program

#include "Vector2D.h"
#include <cassert>
#include <cmath>
#include <iostream>

bool approxEqual(double a, double b, double tol = 1e-9) {
    return std::fabs(a - b) < tol;
}

int main() {
    Vector2D v0;
    assert(approxEqual(v0.x(), 0.0));
    assert(approxEqual(v0.y(), 0.0));

    Vector2D a(3.0, 4.0);
    assert(approxEqual(a.norm(), 5.0));

    Vector2D b(1.0, 2.0);
    Vector2D s = a + b;
    assert(approxEqual(s.x(), 4.0));
    assert(approxEqual(s.y(), 6.0));

    Vector2D d = a - b;
    assert(approxEqual(d.x(), 2.0));
    assert(approxEqual(d.y(), 2.0));

    Vector2D m = b * 3.0;
    assert(approxEqual(m.x(), 3.0));
    assert(approxEqual(m.y(), 6.0));

    Vector2D p1(0.0, 0.0);
    Vector2D p2(3.0, 4.0);
    assert(approxEqual(p1.distanceTo(p2), 5.0));

    std::cout << "All tests passed\n";
}

Why tolerance matters

  • Floating-point arithmetic is approximate
  • Exact equality is often fragile
  • Engineering software usually needs tolerance-based checks
  • Testing strategy should match numerical reality

Tie this directly to measurement uncertainty and numerical reasoning.

Good prompt vs bad prompt

Bad

Write a modern C++ vector class.

Typical outcome:

  • scope drift
  • unrequested features
  • harder idioms
  • weaker pedagogical value

Good

Generate a minimal beginner-friendly Vector2D 
with exactly these operations...

Typical outcome:

  • bounded scope
  • predictable output
  • easier review
  • better alignment with lesson goals

Practical prompting workflow

  1. Write requirements yourself
  2. Ask AI for the interface only
  3. Review and correct the interface
  4. Ask for the implementation only
  5. Review again
  6. Ask for tests
  7. Compile, run, and inspect results
  8. Refactor only what you understand

This is the reusable process students should take away.

Human responsibilities AI cannot take over

  • Choosing requirements
  • Deciding acceptable complexity
  • Checking domain assumptions
  • Numerical validation
  • Integration into the full application
  • Final responsibility

AI can assist with code production, but not with engineering accountability.

Closing checklist

Before AI

  • define requirements
  • define interface
  • state constraints

With AI

  • request one artifact at a time
  • exclude unwanted features
  • keep prompts explicit

After AI

  • review line by line
  • compile with warnings
  • test known cases
  • simplify what you do not understand

End with the professional rule: use AI to accelerate only what you can still inspect, justify, and validate.
